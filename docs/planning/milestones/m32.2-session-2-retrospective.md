### 3. Reusable Session-Expired Pattern Created
**Achievement:** Designed event-based global service that avoids prop-drilling and component coupling.

**Architecture:**
```
SessionExpiredService (singleton)
    ↓ (event: OnSessionExpired)
SessionExpiredModal (subscribed in MainLayout)
    ↓ (captures returnUrl, navigates to /login?returnUrl={url})
Login.razor (handles returnUrl query parameter)
```

**Benefits:**
- **Zero prop-drilling:** Pages call `SessionExpiredService.TriggerSessionExpired()` without knowing about modal
- **Blocking UX:** `MudOverlay` with `ZIndex=9999`, `AutoClose=false` — forces user attention
- **Context preservation:** returnUrl stored in URL (not in-memory) for browser refresh resilience
- **Reusability:** Pattern can be reused for any Blazor WASM project in CritterSupply

**Memory Stored:** "Use event-based SessionExpiredService singleton with blocking modal (not snackbar) for 401 responses. Modal captures returnUrl for post-auth redirect."

## Critical Lessons

### Lesson 1: Optimistic UI Updates Must Handle All Response Codes
**Context:** P0-2 acknowledge button implementation.

**Pattern Used:**
```csharp
_acknowledgingAlerts.Add(alertId);  // Optimistic update
try {
    var response = await client.PostAsync(...);
    if (success) {
        // Remove alert from list immediately
        _alerts = _alerts with { Alerts = ... };
    } else if (409 Conflict) {
        // Already acknowledged by someone else — still remove from list
        _alerts = _alerts with { Alerts = ... };
        Snackbar.Add("Alert was already acknowledged.", Severity.Info);
    } else if (401 Unauthorized) {
        // Session expired — trigger blocking modal
        SessionExpiredService.TriggerSessionExpired();
    }
} finally {
    _acknowledgingAlerts.Remove(alertId);  // Always reset button state
}
```

**Key Insight:** 409 Conflict should **remove the alert from UI** (not just show snackbar) because the alert is acknowledged on the backend. The UX audit requirement "keeps page usable" means non-blocking feedback (Severity.Info snackbar), not "alert stays visible forever".

**Anti-pattern:** Leaving the alert visible after 409 with no way to dismiss it → creates dead UI state.

---

### Lesson 2: Backend Query Filtering Drives Frontend UX Decisions
**Context:** Backend `GetAlertFeed` query filters out acknowledged alerts (`WHERE AcknowledgedBy IS NULL`).

**UX Implication:** Acknowledged alerts **disappear from feed automatically**. This is "acknowledge to dismiss" UX, not "acknowledge to mark as read" UX.

**Frontend Response:** Optimistic update immediately removes alert row (matches eventual backend state after refresh).

**Strategic Decision:** Backend design dictated "remove on acknowledge" UX, so frontend optimistic update must do the same. Misalignment here would cause alert to "flicker back" on next refresh.

**Lesson for Future:** Always check backend query filters when implementing optimistic UI updates — they define the "eventual consistency" target state.

---

### Lesson 3: Event-Based Services Beat HttpMessageHandler for Simple Global State
**Context:** P0-3 session-expired modal trigger mechanism.

**Alternatives Considered:**
1. **Global DelegatingHandler** — Intercepts all HTTP responses, triggers modal on 401
   - **Pros:** Automatic 401 detection, no per-page code
   - **Cons:** Hard to test, couples infrastructure to UI, requires IHttpClientFactory registration gymnastics
2. **Event-based service** — Pages call `TriggerSessionExpired()` explicitly on 401
   - **Pros:** Simple, testable, explicit control flow, no magic
   - **Cons:** Requires per-page code (but only 1 line)

**Decision:** Event-based service (SessionExpiredService) chosen for simplicity and testability.

**Rationale:** Blazor WASM doesn't need the abstraction complexity of DelegatingHandler for 3 pages. The explicit `SessionExpiredService.TriggerSessionExpired()` call is self-documenting and easy to trace.

**Future Consideration:** If Backoffice.Web grows to 20+ API-calling pages, revisit DelegatingHandler approach to reduce boilerplate.

## Strategic Recommendations

### Recommendation 1: Add Global DelegatingHandler in M32.3 (If Write-Ops Expand)
**Rationale:** M32.3 will add write operations (customer notes, return approvals, pricing updates). If we add 10+ new API-calling components, the per-page `SessionExpiredService.TriggerSessionExpired()` pattern becomes boilerplate.

**Proposed Solution:**
```csharp
// Program.cs
builder.Services.AddTransient<SessionExpiredHandler>();
builder.Services.AddHttpClient("BackofficeApi", client => { ... })
    .AddHttpMessageHandler<SessionExpiredHandler>();

// SessionExpiredHandler.cs
public class SessionExpiredHandler : DelegatingHandler {
    protected override async Task<HttpResponseMessage> SendAsync(...) {
        var response = await base.SendAsync(request, ct);
        if (response.StatusCode == HttpStatusCode.Unauthorized) {
            _sessionExpiredService.TriggerSessionExpired();
        }
        return response;
    }
}
```

**Benefit:** Automatic 401 detection for all HTTP clients, no per-page code.

**Trade-off:** Harder to test, but worth it at scale.

### Recommendation 2: Add Playwright E2E Test for Session-Expired Flow (M32.3)
**Rationale:** Session-expired modal is critical UX that must not regress. E2E test should verify:
1. User on `/alerts` page
2. JWT expires (mock via TestContainers time travel or expired token)
3. User clicks "Acknowledge" button → 401 response
4. Session-expired modal appears (blocking overlay)
5. User clicks "Log In Again" → redirected to `/login?returnUrl=/alerts`
6. User logs in → redirected back to `/alerts` page

**File Location:** `tests/Backoffice.E2ETests/Features/SessionExpiredRecovery.feature`

**Implementation:** Use Playwright's `page.goto()`, `page.click()`, `page.waitForSelector()` for modal detection.

**Value:** Prevents regressions in context-preserving re-auth (most important P0-3 requirement).

---

### Recommendation 3: Document "Acknowledge to Dismiss" UX in ADR (M32.3)
**Rationale:** Backend query filters (`WHERE AcknowledgedBy IS NULL`) dictate "acknowledge to dismiss" UX, not "acknowledge to mark as read" UX. This is a **non-obvious architectural decision** that affects frontend behavior.

**Proposed ADR:**
```markdown
# ADR NNNN: Acknowledge-to-Dismiss Alert Feed Design

**Status:** ✅ Accepted

**Decision:** AlertFeedView query filters out acknowledged alerts. Frontend optimistic update removes alert from UI immediately.

**Rationale:** Simpler UX than "acknowledged but still visible" pattern. Reduces UI clutter for operations team.

**Consequences:** Acknowledged alerts are not visible in feed. If user needs audit trail, must query separate "acknowledged alerts" endpoint (future feature).
```

**Benefit:** Documents "why" backend query works this way (not obvious from code alone).

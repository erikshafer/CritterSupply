## 🎓 Key Patterns and Lessons Learned

### 1. BCrypt vs PasswordHasher (Production Alignment)
**Lesson:** Always check production code patterns before writing test code. CritterSupply uses `PasswordHasher<T>` (ASP.NET Core Identity standard) per ADR 0031, not BCrypt.

**Why This Matters:**
- `PasswordHasher<T>` uses PBKDF2-SHA256 with 100,000 iterations (adaptive, secure)
- BCrypt.Net would add unnecessary dependency + misalignment with production auth
- E2E tests should use same password hashing as production for accurate test behavior

**Pattern:**
```csharp
var passwordHasher = new Microsoft.AspNetCore.Identity.PasswordHasher<BackofficeUser>();
var passwordHash = passwordHasher.HashPassword(null!, password);
```

### 2. Wolverine Testing Method Discovery
**Lesson:** Wolverine provides `DisableAllExternalWolverineTransports()` extension method. Don't define duplicate extensions.

**Why This Matters:**
- Wolverine's extension is maintained, tested, and documented
- Duplicate definitions can cause confusion and maintenance burden
- User insight: "DisableAllExternalTransports is something Alba and Wolverine work with" — this is a first-class testing concern

**Pattern:**
```csharp
builder.Host.UseWolverine(opts =>
{
    opts.DisableAllExternalWolverineTransports(); // Wolverine-provided
    opts.Policies.AutoApplyTransactions();
});
```

### 3. Stub Client Signature Alignment
**Lesson:** When creating stub clients for tests, ensure method signatures exactly match expected domain client contracts. Parameter name mismatches cause silent failures or compilation errors.

**Why This Matters:**
- Named parameters in C# require exact name matches
- Parameter name mismatches discovered at compile time (good)
- But can be tedious to fix if not caught early (8 call sites × 3 parameters = 24 individual fixes)

**Pattern:**
```csharp
// Stub client signature (source of truth)
public void AddLowStockAlert(
    Guid alertId,
    string sku,
    int availableQuantity,  // NOT currentStock
    int thresholdQuantity,  // NOT reorderThreshold
    DateTimeOffset createdAt, // NOT triggeredAt
    bool isAcknowledged,
    string severity = "Warning")

// All call sites must match:
stub.AddLowStockAlert(
    id,
    sku,
    availableQuantity: 5,   // ✅ Correct
    thresholdQuantity: 50,  // ✅ Correct
    createdAt: now,         // ✅ Correct
    isAcknowledged: false,
    severity: "Critical");
```

### 4. E2ETestFixture Property Naming Convention
**Lesson:** Use consistent, descriptive property names for server URLs. Prefer `<BC>ApiBaseUrl` pattern.

**Why This Matters:**
- `BackofficeIdentityApiUrl` is ambiguous — is it the full URL or base URL?
- `IdentityApiBaseUrl` and `BackofficeApiBaseUrl` are clearer
- Consistency across fixtures reduces cognitive load

**Pattern:**
```csharp
public sealed class E2ETestFixture
{
    public string IdentityApiBaseUrl { get; private set; } = null!;
    public string BackofficeApiBaseUrl { get; private set; } = null!;
    public string WasmBaseUrl { get; private set; } = null!;
}
```

### 5. 3-Server WASM E2E Pattern
**Lesson:** Blazor WASM E2E tests require orchestrating 3 separate servers (JWT issuer, BFF API, static WASM host). This is distinct from Blazor Server (1 server) or traditional SPA (2 servers: API + static host).

**Why This Matters:**
- JWT issuer (BackofficeIdentity.Api) must be separate for realistic auth flow
- BFF (Backoffice.Api) consumes JWT, aggregates domain BCs
- Static WASM host (Backoffice.Web) serves index.html + Blazor WASM runtime

**Architecture:**
```
┌─────────────────────┐
│ Backoffice.Web      │ ← Static WASM host (port 0 → random)
│ (wwwroot/index.html)│
└──────────┬──────────┘
           │ HTTP (JWT Bearer)
           ▼
┌─────────────────────┐
│ Backoffice.Api      │ ← BFF (aggregates Orders, Returns, Inventory, etc.)
│ (port 0 → random)   │
└──────────┬──────────┘
           │ HTTP (login, token refresh)
           ▼
┌─────────────────────┐
│ BackofficeIdentity  │ ← JWT issuer (login, token issuance)
│ .Api (port 0)       │
└─────────────────────┘
```

### 6. Stub Data Coordination
**Lesson:** Use `WellKnownTestData` constants for deterministic GUIDs/emails. Prevents coordination issues when step definitions reference same entities.

**Why This Matters:**
- Random GUIDs per scenario cause "entity not found" errors
- Deterministic GUIDs allow "Given user X exists" → "When I search for X" → "Then I see X's details"

**Pattern:**
```csharp
public static class WellKnownTestData
{
    public static class Customers
    {
        public static readonly Guid Alice = Guid.Parse("11111111-1111-1111-1111-111111111111");
        public static readonly Guid Bob = Guid.Parse("22222222-2222-2222-2222-222222222222");
    }
}
```

## 🚧 Challenges and How They Were Overcome

### Challenge 1: BCrypt Compilation Error
**Problem:** Test code used `BCrypt.Net.BCrypt.HashPassword()` but BCrypt.Net-Next package wasn't referenced. Build failed with CS0103.

**Investigation:**
- Used Task agent to search for BCrypt usage in production code
- Found zero results
- Searched for "password hash" patterns in BackofficeIdentity
- Discovered `PasswordHasher<BackofficeUser>` is production standard

**Resolution:**
- Replaced BCrypt with `PasswordHasher<BackofficeUser>`
- Matches production authentication (ADR 0031: PBKDF2-SHA256)
- No additional dependencies required

**Takeaway:** Always check production code patterns before writing test code. Don't assume common libraries (like BCrypt) are in use without verification.

### Challenge 2: Duplicate Wolverine Extension Method
**Problem:** E2ETestFixture defined custom `DisableAllExternalWolverineTransports` extension, but Wolverine already provides this method. Compilation error.

**User Insight:** "DisableAllExternalTransports is something Alba and Wolverine work with such a method name to allow for smoother types of tests."

**Resolution:**
- Removed duplicate extension method definition (lines 501-507)
- Use Wolverine's built-in `DisableAllExternalWolverineTransports()` extension

**Takeaway:** When working with established frameworks like Wolverine, check for built-in testing utilities before defining custom extensions. The framework authors often provide first-class testing support.

**Resolution:**

- Fixed all 8 occurrences of `reorderThreshold` → `thresholdQuantity`
- Fixed all 8 occurrences of `triggeredAt` → `createdAt`

**Takeaway:** When creating stub clients, define the signature first, then write step definitions. Reduces rework.

### Challenge 4: Fixture Property Name Mismatches
**Problem:** AuthenticationSteps used `BackofficeIdentityApiUrl` and `BackofficeApiUrl`, but E2ETestFixture defined `IdentityApiBaseUrl` and `BackofficeApiBaseUrl`.

**Resolution:**

**Takeaway:** Establish naming conventions for fixture properties early. Prevents refactoring churn.

## 🔄 What Would We Do Differently Next Time?

### 1. Define Stub Client Signatures Earlier
**Issue:** Many parameter name mismatches in Session 9b.

**Better Approach:**
1. Define stub client interface/signature first (e.g., `StubInventoryClient.AddLowStockAlert`)
2. Write BDD scenarios second (reference stub client methods)
3. Write step definitions third (call stub client with correct parameter names)

**Benefit:** Reduces compilation error churn from 24 fixes → 0 fixes.

### 2. Verify Production Auth Pattern Before Implementing Test Auth
**Issue:** Test code assumed BCrypt, but production uses `PasswordHasher<T>`.

**Better Approach:**
1. Search production codebase for password hashing patterns (`grep -r "password" src/Backoffice\ Identity/`)
2. Verify ADR/CONTEXTS.md for authentication decisions (ADR 0031)
3. Implement test auth using same pattern as production

**Benefit:** Eliminates BCrypt compilation error, ensures test/production alignment.

### 3. Use Playwright Codegen for Initial Page Object Model Selectors
**Issue:** Manually writing `data-testid` selectors is tedious and error-prone.

**Better Approach:**
- Use `playwright codegen http://localhost:5244` to record interactions
- Codegen generates selector suggestions (including MudBlazor MudSelect `.mud-input-slot` patterns)
- Refactor codegen output into Page Object Model methods

**Benefit:** Faster Page Object Model creation, fewer selector mistakes.

### 4. Check Wolverine Documentation for Testing Utilities
**Issue:** Duplicate `DisableAllExternalWolverineTransports` extension method.

**Better Approach:**
- Search Wolverine docs/GitHub for "DisableAllExternal" or "testing"
- Check Alba integration docs for Wolverine-specific testing methods
- Use framework-provided utilities instead of custom extensions

**Benefit:** Eliminates duplicate code, leverages maintained framework utilities.

## 📚 Key Decisions

### D1: Tiered Timeout Strategy for WASM E2E Tests

**Context:** Blazor WASM E2E tests have unpredictable timing due to:
1. WASM download + hydration (5-30s)
2. MudBlazor component initialization (2-5s)
3. JWT auth state propagation (1-3s)
4. SignalR connection establishment (1-5s)

**Decision:** Use tiered timeout strategy:
- **Initial page load:** 15s + explicit MudBlazor provider check
- **Authenticated navigation:** 15s (auth state propagation)
- **Element visibility:** 10s (standard)
- **SignalR connection:** 15s (JWT dependency)
- **State checks:** 5s (DOM polling)

**Rationale:** Fixed timeouts (30s everywhere) are too aggressive for initial load but wasteful for fast operations. Tiered timeouts optimize for actual operation timing.

**Alternatives Considered:**
1. **Single 30s timeout everywhere** — Wasteful for fast operations
2. **Playwright's auto-wait only** — Insufficient for WASM hydration
3. **Exponential backoff** — Overcomplicated for E2E tests

---

### D2: Explicit MudBlazor Hydration Detection

**Context:** WASM tests were timing out because MudBlazor hadn't finished initializing even after `NetworkIdle`.

**Decision:** Add explicit check for MudBlazor provider CSS class:
```csharp
await _page.WaitForSelectorAsync(".mud-dialog-provider",
    new() { State = WaitForSelectorState.Attached, Timeout = 15_000 });
```

**Rationale:** MudBlazor v9+ creates `.mud-dialog-provider` and `.mud-snackbar-provider` containers during initialization. Waiting for these ensures MudBlazor is ready.

**Why This Works:** MudBlazor providers are created in `MudProviders` component, which is the last step of MudBlazor initialization. If provider exists, MudBlazor is ready.

## 📊 Lessons Learned

### L1: WASM E2E Tests Need Generous Initial Timeouts

**Discovery:** Blazor WASM cold start can take 5-30s depending on:
- Browser cache state
- .NET runtime download
- `_framework` files size
- CPU speed
- Network latency

**Pattern:**
```csharp
// ❌ WRONG - Too aggressive for WASM
await EmailInput.WaitForAsync(new() { Timeout = 5_000 });

// ✅ RIGHT - Tiered timeout with explicit hydration check
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);
await _page.WaitForSelectorAsync(".mud-dialog-provider", new() { Timeout = 15_000 });
await EmailInput.WaitForAsync(new() { Timeout = 15_000 });
```

**Why:** WASM apps are fundamentally different from server-rendered apps. Initial load includes downloading/compiling .NET runtime + app assemblies.

---

### L2: Explicit Hydration Checks Prevent Flaky Tests

**Problem:** `NetworkIdle` doesn't guarantee MudBlazor components are ready. Tests were clicking elements before MudBlazor event handlers attached.

**Solution:** Check for framework-specific markers (MudBlazor provider classes, SignalR connection state, auth state).

**Pattern:**
```csharp
// Step 1: Wait for network idle
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);

// Step 2: Wait for framework initialization
await _page.WaitForSelectorAsync(".mud-dialog-provider", new() { Timeout = 15_000 });

// Step 3: Wait for specific element
await EmailInput.WaitForAsync(new() { Timeout = 15_000 });
```

**Why:** Each step verifies a different layer of readiness. Skipping any step causes flaky tests.

---

### L3: Auth State Propagation Needs Extra Time in WASM

**Discovery:** After successful login API call, Blazor WASM takes 1-3s to:
1. Update `BackofficeAuthState`
2. Fire `OnChange` event
3. Call `NotifyAuthenticationStateChanged()`
4. Re-render protected components
5. Execute `NavigationManager.NavigateTo("/dashboard")`

**Impact:** 10s timeout for dashboard navigation was insufficient. 15s is more reliable.

**Pattern:**
```csharp
await LoginButton.ClickAsync();

// Wait for auth state propagation + navigation (15s instead of 10s)
await _page.WaitForURLAsync(url => url.Contains("/dashboard"), new() { Timeout = 15_000 });
```

---

### L4: SignalR Connection Depends on JWT Auth Completion

**Discovery:** SignalR hub uses `AccessTokenProvider` delegate to get JWT from `BackofficeAuthState`. If auth state isn't ready, connection fails silently.

**Impact:** SignalR connection timeout must be >= auth state propagation timeout.

**Pattern:**
```csharp
// After login, wait for auth state propagation
await _page.WaitForURLAsync(url => url.Contains("/dashboard"), new() { Timeout = 15_000 });

// Then wait for SignalR connection (15s to allow for retry attempts)
await RealtimeIndicator.WaitForAsync(new() { Timeout = 15_000 });
```

**Why:** SignalR client retries connection on failure. If JWT isn't ready on first attempt, retry succeeds after auth state propagates.

### R5: Consider Playwright's `page.waitForLoadState('networkidle')` + `domcontentloaded`

**Why:** Current pattern uses `NetworkIdle` but doesn't wait for `domcontentloaded`. This may cause timing issues.

**Pattern:**
```csharp
// Current
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);

// Enhanced
await _page.WaitForLoadStateAsync(LoadState.DOMContentLoaded);
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);
```

**Rationale:** `DOMContentLoaded` fires when HTML is parsed but before all resources load. `NetworkIdle` fires when network activity stops. Waiting for both ensures DOM is ready AND resources loaded.

**Impact:** May reduce flaky tests caused by clicking elements before DOM is ready.

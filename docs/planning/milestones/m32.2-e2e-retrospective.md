## Technical Highlights

### 1. SimulateSessionExpired Pattern for Stub Clients

**Problem:** Need to test session expiry (401 Unauthorized) responses without spinning up real auth infrastructure in E2E tests.

**Solution:** Added `SimulateSessionExpired` bool property to all 3 stub clients. When true, throw `HttpRequestException` with `HttpStatusCode.Unauthorized`.

**Why This Works:**
- ✅ In-process exception (no HTTP layer mocking)
- ✅ Triggers SessionExpiredService event globally (same as production 401 responses)
- ✅ Works across all authenticated API calls

**Implementation:**
```csharp
public bool SimulateSessionExpired { get; set; }

public Task<StockLevelDto?> GetStockLevelAsync(string sku, CancellationToken ct = default)
{
    if (SimulateSessionExpired)
        throw new HttpRequestException("Session expired", null, HttpStatusCode.Unauthorized);

    return Task.FromResult(_stockLevels.GetValueOrDefault(sku));
}
```

**Step Definition Usage:**
```csharp
[When(@"my session expires")]
public void WhenMySessionExpires()
{
    Fixture.StubInventoryClient.SimulateSessionExpired = true;
    Fixture.StubOrdersClient.SimulateSessionExpired = true;
    Fixture.StubCustomerIdentityClient.SimulateSessionExpired = true;
}
```

### 2. Multi-Role Admin User Seeding

**Problem:** Authorization tests need deterministic admin users with specific roles (7 roles: system-admin, operations-manager, warehouse-clerk, customer-service, copy-writer, pricing-manager, executive).

**Solution:** Extended E2ETestFixture with `SeedAdminUserWithRole()` method + WellKnownTestData constants for all 7 roles.

**Why This Works:**
- ✅ Deterministic GUIDs (repeatable tests)
- ✅ Role-specific permissions enforced by Backoffice Identity BC
- ✅ Uses production PasswordHasher (PBKDF2-SHA256)

**Implementation:**
```csharp
public void SeedAdminUserWithRole(Guid userId, string email, string displayName, string role)
{
    var passwordHasher = new PasswordHasher<BackofficeUser>();

    var backofficeRole = role switch
    {
        "system-admin" => BackofficeRole.SystemAdmin,
        "operations-manager" => BackofficeRole.OperationsManager,
        "warehouse-clerk" => BackofficeRole.WarehouseClerk,
        "customer-service" => BackofficeRole.CustomerService,
        "copy-writer" => BackofficeRole.CopyWriter,
        "pricing-manager" => BackofficeRole.PricingManager,
        "executive" => BackofficeRole.Executive,
        _ => throw new ArgumentException($"Unknown role: {role}")
    };

    var user = new BackofficeUser(userId, email, displayName, backofficeRole, IsActive: true);
    user.PasswordHash = passwordHasher.HashPassword(user, "Password123!");

    _backofficeIdentityDbContext.BackofficeUsers.Add(user);
    _backofficeIdentityDbContext.SaveChanges();
}
```

**Step Definition Usage:**
```csharp

    await loginPage.NavigateAsync();
    await loginPage.LoginAsync(email, "Password123!");
```

### 4. Playwright `.First` Property (Not Method)

**Problem:** Used LINQ-style `.First()` method but Playwright uses `.First` property.

**Error:**
```
CS1061: 'ILocator' does not contain a definition for 'First' and no accessible extension method 'First' accepting a first argument of type 'ILocator' could be found
```

**Root Cause:** Playwright's `ILocator` interface uses properties, not methods, for indexing.

**Wrong:**
```csharp
var firstAlert = AlertRows.First();  // ❌ Method call
```

**Right:**
```csharp
var firstAlert = AlertRows.First;  // ✅ Property access
```

**Key Lesson:** Playwright uses properties for indexing, not LINQ methods. Use `.First`, `.Last` (properties) and `.Nth(n)` (method with index parameter).

### 6. ReturnUrl Preservation Testing

**Problem:** After session expiry, user should be redirected back to original page after re-authentication.

**Solution:** Verify `returnUrl` query parameter is preserved and page is restored post-login.

**Key Assertions:**
1. `returnUrl` query parameter present and correctly URL-encoded
2. Post-login redirect lands on original page
3. Page-specific elements visible (proves full navigation completed)

**Implementation:**
```csharp
[Then(@"the returnUrl query parameter should be ""(.*)""")]
public async Task ThenTheReturnUrlQueryParameterShouldBe(string expectedReturnUrl)
{
    await Task.Delay(500); // Allow navigation to complete
    var url = Page.Url;
    url.ShouldContain($"returnUrl={Uri.EscapeDataString(expectedReturnUrl)}");
}

[Then(@"I should be redirected back to the dashboard")]
public async Task ThenIShouldBeRedirectedBackToTheDashboard()
{
    var dashboardPage = new DashboardPage(Page, Fixture.WasmBaseUrl);
    var isOnDashboard = await dashboardPage.IsOnDashboardPageAsync();
    isOnDashboard.ShouldBeTrue();
}
```

**Why This Matters:**
- ✅ User returns to exact page they were working on
- ✅ No lost context (filters, scroll position preserved by page state)
- ✅ Validates navigation guard + returnUrl wiring

## Recommendations for Future E2E Work

### 1. Establish Enum Verification Checklist
When adding new role-based tests:
1. ✅ Grep production enum definition FIRST
2. ✅ Document all enum values in WellKnownTestData
3. ✅ Map kebab-case strings to actual enum values in switch statement
4. ✅ Run `dotnet build` to verify mappings before writing tests

### 2. Playwright API Pattern Reference
Create a quick reference card for common Playwright gotchas:
- `.First`, `.Last` → properties (not methods)
- `.Nth(n)` → method with index parameter
- `.Filter(new() { HasText = "..." })` → method chaining pattern
- `.WaitForAsync(new() { State = ..., Timeout = ... })` → async waiting with options

### 3. Global Page Object Guidelines
Document when to create global vs page-specific page objects:
- **Global:** Modals, navigation bars, footer, toast notifications (appear across multiple pages)
- **Page-Specific:** Forms, tables, filters (tied to single page/feature)

### 4. Stub Client 401 Simulation Pattern
Standardize across all E2E test projects:
- All stub clients should have `SimulateSessionExpired` property
- All authenticated methods check this flag first
- Throw `HttpRequestException` with `HttpStatusCode.Unauthorized`
- Document in `e2e-playwright-testing.md`

### 5. Test Data Determinism
- Always use WellKnownTestData for deterministic GUIDs
- Avoid random IDs in E2E tests (breaks repeatability)
- Use kebab-case role strings ("system-admin", "pricing-manager")

### 6. Build Verification Workflow
- Run `dotnet build` after EVERY step definition file is created
- Fix compile errors immediately (don't batch fixes)
- Build succeeds = 0 errors (warnings OK if pre-existing)

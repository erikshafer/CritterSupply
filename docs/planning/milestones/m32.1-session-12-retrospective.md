## What Worked

### 1. Systematic Debugging with Server-Side Logging

**Context**: Session 11 ended with mysterious 404 errors visible in browser console but no details about which resources were failing.

**Actions**:
- Added comprehensive request/response logging middleware to `WasmStaticFileHost`
- Logged all incoming requests with method, path, querystring, host
- Logged all outgoing responses with status codes
- Added special logging for 404 responses
- Added `OnPrepareResponse` callback to log static file serving

**Result**: Immediately identified that `/index.html` was returning 404, which led directly to root cause.

**Code**:
```csharp
// Add comprehensive request/response logging middleware BEFORE any other middleware
_app.Use(async (context, next) =>
{
    Console.WriteLine($"[WasmStaticFileHost] → Request: {context.Request.Method} {context.Request.Path}{context.Request.QueryString}");
    Console.WriteLine($"[WasmStaticFileHost]   Host: {context.Request.Host}");

    await next();

    Console.WriteLine($"[WasmStaticFileHost] ← Response: {context.Response.StatusCode} for {context.Request.Method} {context.Request.Path}");

    if (context.Response.StatusCode == 404)
    {
        Console.WriteLine($"❌ [WasmStaticFileHost] 404 NOT FOUND: {context.Request.Method} {context.Request.Path}");
    }
});
```

### 2. Correct ASP.NET Core Middleware Ordering

**Context**: `MapGet("/appsettings.json")` was called BEFORE `UseStaticFiles`, which meant route handlers processed requests before static file middleware.

**Fix**: Reordered middleware to follow ASP.NET Core conventions:
1. `UseCors()`
2. Request/response logging middleware
3. **`UseStaticFiles` (FIRST for serving files)**
4. `MapGet("/appsettings.json")` (override specific file)
5. `MapFallbackToFile("index.html")` (SPA fallback)

**Why this matters**: In ASP.NET Core, middleware order is critical. Static files should be served before route handlers to avoid unnecessary routing overhead.

**Commit**: `6c72f78` — "Fix middleware ordering (UseStaticFiles before route handlers)"

### 3. Root Cause Discovery: Missing `index.html` in Build Output

**Investigation**:
- Checked `src/Backoffice/Backoffice.Web/bin/Debug/net10.0/wwwroot/` — only had `_framework` directory, **no `index.html`**
- Checked source `src/Backoffice/Backoffice.Web/wwwroot/` — `index.html` present
- Confirmed `index.html` is only created by `dotnet publish`, not `dotnet build`

**Root Cause**: Blazor WASM SDK behavior:
- **`dotnet build`** — Compiles assemblies to `_framework` but doesn't copy `wwwroot` content (like `index.html`, `appsettings.json`)
- **`dotnet publish`** — Creates complete deployable output with all wwwroot files

**Fix**: Updated `FindWasmRoot()` to prioritize publish output directory:
1. Check `bin/Debug/net10.0/publish/wwwroot` (publish output — complete app)
2. Check `bin/Debug/net10.0/wwwroot` (build output — incomplete)
3. Check source `wwwroot` (last resort)

**Code**:
```csharp
// PRIORITY 1: Check publish output directory (has index.html + _framework)
var publishWwwroot = Path.Combine(current, "src", "Backoffice", "Backoffice.Web", "bin", "Debug", "net10.0", "publish", "wwwroot");

{
    Console.WriteLine($"✅ [FindWasmRoot] Found publish wwwroot with _framework and index.html: {publishWwwroot}");
    return publishWwwroot;
}
```

## Lessons Learned

### 1. Middleware Ordering Matters in ASP.NET Core

**Anti-pattern**:
```csharp
app.MapGet("/appsettings.json", ...);  // Route handler FIRST
app.UseStaticFiles();                  // Static files SECOND
```

**Correct pattern**:
```csharp
app.UseStaticFiles();                  // Static files FIRST
app.MapGet("/appsettings.json", ...);  // Route handler overrides SECOND
```

**Why**: Route handlers bypass static file middleware if they match first. Static files should always be served first for performance and correctness.

### 2. `dotnet build` vs `dotnet publish` for Blazor WASM

**Key difference**:
- **`dotnet build`** — Development-focused, compiles assemblies, skips wwwroot file copy
- **`dotnet publish`** — Deployment-focused, creates complete output with all assets

**For E2E tests**: Always use `dotnet publish` output, not `dotnet build` output, because E2E tests simulate real deployment scenarios.

**Documentation update needed**: E2E test README should document that `dotnet publish` is required before running tests.

### 3. Comprehensive Logging is Essential for Debugging

Without the request/response logging middleware, we would have spent hours guessing which file was missing. The logging revealed the issue immediately:

```
[WasmStaticFileHost] → Request: GET /login
[WasmStaticFileHost] ← Response: 404 for GET /index.html
❌ [WasmStaticFileHost] 404 NOT FOUND: GET /index.html
```

**Pattern to remember**: When debugging web apps, always log:
1. All incoming requests (method, path, host)
2. All outgoing responses (status code)
3. Special markers for error conditions (404, 500, etc.)

### 4. Browser Console Errors Provide Critical Clues

The browser console showed:
- **Session 11**: Generic "Failed to load resource: 404" (no URL)
- **Session 12**: Specific authorization policy errors with stack traces

**Pattern**: Always capture browser console output in E2E tests using Playwright's `page.Console` event handler (already implemented in `TestHooks.cs`).

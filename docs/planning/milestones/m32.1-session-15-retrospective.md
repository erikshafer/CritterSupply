## 📝 Test-ID Naming Conventions (Documented)

**Convention Established:**

| Element Type | Pattern | Examples |
|--------------|---------|----------|
| KPI Cards | `kpi-{metric-name}` | `kpi-total-orders`, `kpi-revenue`, `kpi-pending-returns` |
| KPI Values (nested) | `kpi-value` | Always nested within KPI card element |
| Navigation Links | `nav-{destination}` | `nav-customer-service`, `nav-operations`, `nav-analytics` |
| Form Inputs | `{form}-{field}` | `login-email`, `login-password` |
| Form Buttons | `{form}-{action}` | `login-submit`, `logout-button` |
| Real-time Indicators | `realtime-{state}` | `realtime-connected`, `realtime-disconnected` |

**Anti-Patterns to Avoid:**
- ❌ Generic names: `button1`, `div-content`, `label-text`
- ❌ Component-specific names: `mud-button`, `mud-card` (these are CSS classes, not test-ids)
- ❌ Presentational names: `red-banner`, `large-font` (use semantic names)
- ❌ Index-based names: `kpi-1`, `alert-2` (fragile, breaks on reordering)

**Rationale:**
- **Semantic names** describe what the element represents, not how it looks
- **Kebab-case** aligns with HTML attribute conventions
- **Hierarchical structure** (parent-child) makes locators more robust
- **Stable identifiers** don't change when UI styling or ordering changes

**Reference:** See `docs/skills/e2e-playwright-testing.md` (updated in Session 15)

## 🔍 Lessons Learned

### L1: E2E Test Timeouts Are Multifaceted

**Discovery:** E2E test timeouts are rarely caused by a single issue. Multiple async operations compound: WASM download → Blazor hydration → MudBlazor init → JWT auth → SignalR connection → API calls → UI rendering.

**Impact:** 10-second timeout may be insufficient for full workflow in CI environment (slower CPU, network latency).

**Recommendation:** Use hierarchical timeouts: 30s for initial page load, 10s for navigation, 5s for individual element visibility.

### L2: Page Object Models Should Match Implementation Reality

**Discovery:** DashboardPage.cs included `ActiveCustomersCard` that was never implemented in Dashboard.razor. This created a mismatch between test expectations and actual UI.

**Impact:** Tests fail with "element not found" errors even though underlying functionality works correctly.

**Recommendation:** Keep Page Object Models synchronized with UI implementation. Remove deferred features from tests until actually implemented.

### L3: Playwright Tracing Is Essential for Diagnosing Timeouts

**Discovery:** Without Playwright trace files (`.zip` artifacts), diagnosing E2E failures is guesswork. Traces capture screenshots, DOM snapshots, network traffic, and console logs.

**Impact:** Session 14 test failure had truncated output; we couldn't see actual error without trace files.

**Recommendation:** Always enable Playwright tracing in CI. Store traces as artifacts for failed test runs. Use `await context.Tracing.StartAsync()` in `BeforeScenario` hook (already done).

### L4: Test-ID Naming Conventions Prevent Refactoring Breakage

**Discovery:** Session 14 had 17 test-id mismatches because Page Object Model used inconsistent naming (`kpi-active-orders` vs `kpi-total-orders`, `customer-search-btn` vs `nav-customer-service`).

**Impact:** Tests failed not because functionality broke, but because test-ids didn't match POM locators.

**Recommendation:** Document test-id conventions early (Session 15). Enforce conventions via PR review checklist. Use hierarchical naming (parent-child) for nested elements.

### L5: Blazor WASM E2E Tests Need Generous Timeouts

**Discovery:** Blazor WASM has unpredictable cold start times (5-30s) depending on: browser cache state, .NET runtime download, `_framework` files size, and CPU speed.

**Impact:** 5-second default timeouts are too aggressive for WASM E2E tests. 30-second timeout for initial page load is more realistic.

**Recommendation:** Use tiered timeouts: 30s for `/login` navigation, 15s for authenticated navigation, 10s for element visibility, 5s for DOM state checks.

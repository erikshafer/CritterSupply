## 🐛 Root Cause Analysis

### Primary Issue: All Tests Timing Out (~30 seconds)

**Symptoms:**
1. Every test waits for elements that never appear
2. Consistent 30-second timeout (Playwright default)
3. No browser console errors captured (tracing not enabled)
4. Server logs show APIs responding correctly

**Likely Root Causes (Ordered by Probability):**

#### 1. **Blazor WASM Hydration Failure (Most Likely)**

**Evidence:**
- All tests fail at first element wait (`Page.GetByTestId(...)`)
- WASM apps require hydration before interactive
- No explicit wait for `DOMContentLoaded` or `NetworkIdle` in test fixture

**Hypothesis:** Blazor WASM is loading but not hydrating, leaving DOM in non-interactive state.

**Next Steps:**
- Enable Playwright tracing to capture browser console logs
- Add explicit wait: `await Page.WaitForLoadStateAsync(LoadState.NetworkIdle)`
- Check for JavaScript errors in browser console (via tracing)
- Verify `blazor.webassembly.js` loads successfully

#### 2. **Appsettings.json Injection Failure (High Probability)**

**Evidence:**
- `WasmStaticFileHost.InjectTestConfig()` relies on `wwwroot/appsettings.json` interception
- WASM needs API URLs (BackofficeApiBaseUrl, IdentityApiBaseUrl) to function
- If injection fails, WASM can't make HTTP calls → hangs

**Hypothesis:** appsettings.json not being intercepted/injected correctly, WASM lacks API URLs.

**Next Steps:**
- Add logging to `WasmStaticFileHost` to verify interception
- Check browser network tab (via tracing) for 404s or CORS errors
- Verify `appsettings.json` contains correct test URLs

#### 3. **Page Object Model Selector Mismatches (Medium Probability)**

**Evidence:**
- Tests use `data-testid` attributes
- Production Blazor components may not have `data-testid` attributes yet

**Hypothesis:** `data-testid` attributes don't exist in rendered HTML.

**Next Steps:**
- Run test with headed browser (`Headless = false`) to inspect DOM
- Add `data-testid` attributes to Blazor components
- Use Playwright Inspector to verify selectors

#### 4. **SignalR Connection Blocking (Low Probability)**

**Evidence:**
- BackofficeHub configured with antiforgery disabled
- SignalR connection happens after login

**Hypothesis:** SignalR connection attempt blocks WASM hydration.

**Next Steps:**
- Disable SignalR in test environment
- Check browser console for SignalR connection errors

## 🎓 Key Lessons Learned

### L1: Infrastructure ≠ Application Readiness
**Lesson:** 3-server WASM pattern can start successfully (servers respond to health checks) but application may not be functional.

**Why This Matters:**
- TestContainers reports "ready" when PostgreSQL accepts connections
- Kestrel reports "ready" when servers bind to ports
- Neither indicates Blazor WASM is hydrated and interactive

**Pattern for Next Session:**
```csharp
// Add to E2ETestFixture
await Page.WaitForLoadStateAsync(LoadState.NetworkIdle);
await Page.WaitForSelectorAsync("[data-testid='app-loaded']", new() { State = WaitForSelectorState.Visible });
```

### L2: Playwright Tracing is Essential for E2E Debugging
**Lesson:** Without tracing, E2E test failures are opaque. You can't see browser console errors, network traffic, or DOM state.

**Why This Matters:**
- Timeout errors provide zero diagnostic information
- Browser console logs reveal JavaScript errors
- Network tab shows 404s, CORS errors, failed API calls
- Screenshots show actual DOM state vs expected

**Pattern for Next Session:**
```csharp
// Enable tracing in E2ETestFixture
await Browser.StartTracingAsync(new()
{
    Screenshots = true,
    Snapshots = true,
    Sources = true
});

// On test failure, save trace
await Browser.StopTracingAsync(new() { Path = $"traces/test-{scenarioTitle}.zip" });
```

### L3: Headed Browser for Initial E2E Test Runs
**Lesson:** Headless browser hides critical information during initial test development. Run headed first, then switch to headless for CI.

**Why This Matters:**
- Can visually inspect WASM hydration progress
- Can see "Loading..." indicators stuck
- Can open DevTools and check console/network manually

**Pattern for Next Session:**
```csharp
// E2ETestFixture: Make Headless configurable
var headless = Environment.GetEnvironmentVariable("PLAYWRIGHT_HEADLESS") != "false";
Browser = await Playwright.Chromium.LaunchAsync(new() { Headless = headless });
```

Run with: `PLAYWRIGHT_HEADLESS=false dotnet test`

### L4: WASM Static File Interception Requires Verification
**Lesson:** `WasmStaticFileHost.InjectTestConfig()` is a custom pattern for injecting test URLs into appsettings.json. Must verify it works before assuming WASM has correct config.

**Why This Matters:**
- WASM can't discover API URLs at runtime (no server-side config)
- If appsettings.json injection fails, WASM hangs on first HTTP call
- Failure is silent (no error, just timeout)

**Pattern for Next Session:**
```csharp
// Add logging to WasmStaticFileHost
_logger.LogInformation("Intercepting appsettings.json request");
_logger.LogInformation("Injected config: IdentityApiBaseUrl={Identity}, BackofficeApiBaseUrl={Backoffice}",
    config.IdentityApiBaseUrl, config.BackofficeApiBaseUrl);
```

### L5: E2E Test Execution Time Budget
**Lesson:** 32 scenarios × 30 seconds timeout = 16 minutes minimum execution time. This is unacceptable for CI.

**Why This Matters:**
- CI builds will timeout (most CI has 10-15 minute job limits)
- Developer feedback loop too slow
- Need to reduce timeout or parallelize

**Options for Next Session:**
1. **Reduce timeout:** `Page.SetDefaultTimeout(10_000)` (10 seconds) after confirming WASM loads
2. **Parallelize:** `dotnet test --parallel` (requires fixture changes)
3. **Selective runs:** Run only auth tests first, then customer service, then ops alerts

## 🚧 Challenges and Blockers

### C1: No Playwright Tracing Configured
**Problem:** Can't diagnose root cause without browser console logs, network traffic, or screenshots.

**Impact:** Blocked on identifying whether issue is WASM hydration, appsettings injection, selector mismatch, or SignalR.

**Resolution for Session 11:**
- Add `Browser.StartTracingAsync()` to `BeforeScenario` hook
- Add `Browser.StopTracingAsync()` to `AfterScenario` hook (on failure)
- Upload trace files as CI artifacts

### C2: No data-testid Attributes in Production Components
**Problem:** Tests assume `data-testid` attributes exist, but may not be implemented in Blazor components yet.

**Impact:** Even if WASM hydrates, tests will fail if selectors don't match.

**Resolution for Session 11:**
- Run headed browser to inspect rendered DOM
- Add `data-testid` attributes to critical components (login form, dashboard, customer search)
- Document attribute naming convention

### C3: No WASM Load Indicator
**Problem:** Tests don't wait for WASM to fully load before interacting.

**Impact:** Tests may interact with DOM before Blazor has hydrated, causing failures.

**Resolution for Session 11:**
- Add `<div data-testid="app-loaded" style="display:none"></div>` to `App.razor` after hydration
- Wait for indicator in `BeforeScenario` hook

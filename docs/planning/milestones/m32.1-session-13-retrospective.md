### 3. JWT Role Claim Format (Kebab-Case)
**Fixed:** JWT issued PascalCase role claims (`"SystemAdmin"`) but authorization policies expected kebab-case (`"system-admin"`)
**Impact:** Authorization checks would fail even with valid JWT

**Solution:**
Created `BackofficeRoleExtensions.ToRoleString()` method:
```csharp
public static string ToRoleString(this BackofficeRole role)
{
    return PascalToKebabCaseRegex()
        .Replace(role.ToString(), "-$1")
        .Trim('-')
        .ToLowerInvariant();
}
```

Updated `JwtTokenGenerator`:
```csharp
new Claim(ClaimTypes.Role, user.Role.ToRoleString())  // "system-admin" instead of "SystemAdmin"
```

## 📚 Lessons Learned

### 1. Blazor WASM Authorization Requires Client-Side Policy Registration
**Problem:** In Blazor Server, authorization policies are server-side only. In WASM, `<AuthorizeView Policy="...">` requires `AddAuthorizationCore(opts => opts.AddPolicy(...))` on the client.

**Pattern:**
```csharp
// Server-side (Backoffice.Api/Program.cs)
builder.Services.AddAuthorizationBuilder()
    .AddPolicy("Executive", policy => policy.RequireRole("executive"));

// Client-side (Backoffice.Web/Program.cs) — MUST ALSO REGISTER
builder.Services.AddAuthorizationCore(opts =>
{
    opts.AddPolicy("Executive", policy => policy.RequireRole("executive"));
});
```

**Why:** WASM authorization is entirely client-side. Blazor reads JWT claims and evaluates policies locally.

---

### 2. JWT Role Claims Must Use Kebab-Case for .NET Authorization Policies
**Problem:** .NET authorization checks are case-sensitive and match policy role names exactly.

**Convention:** Use kebab-case for multi-word roles:
- `"system-admin"` not `"SystemAdmin"` or `"SYSTEM_ADMIN"`
- `"operations-manager"` not `"OperationsManager"`
- `"cs-agent"` not `"CSAgent"` or `"CustomerService"`

**Implementation:**
Create extension method to convert enum to kebab-case:
```csharp
public static string ToRoleString(this BackofficeRole role)
{
    return PascalToKebabCaseRegex()
        .Replace(role.ToString(), "-$1")
        .Trim('-')
        .ToLowerInvariant();
}
```

---

### 3. Consistent Test-ID Naming Between Page Objects and Components
**Problem:** Page Object Models define expected `data-testid` values, but Razor components must actually render them.

**Best Practice:**
1. Write Page Object Model first (defines contract)
2. Add `data-testid` attributes to Razor components matching POM expectations
3. Use consistent naming:
   - Form inputs: `{page}-{field}` (e.g., `login-email`, `login-password`)
   - Buttons: `{page}-{action}` (e.g., `login-submit`, `logout-button`)
   - Alerts: `{page}-{type}` (e.g., `login-error`, `login-loading`)
   - KPI cards: `kpi-{metric-name}` (e.g., `kpi-total-orders`)

---

### 4. WASM Publishing is Required for E2E Tests
**Problem:** `dotnet build` produces intermediate outputs. E2E tests need the final published WASM assets.

**Solution:**
```bash
dotnet publish "src/Backoffice/Backoffice.Web" -c Debug
```

**Why:** Publishing triggers:
- WASM IL linking/trimming
- `wwwroot/` assembly of all static assets
- `index.html` generation with correct `_framework` references

**Note:** This was already implemented correctly in Session 12.

## 🔧 Core Problem Solved

### Symptom
OrderNote events were being appended successfully (201 responses), but queries returned empty results (count = 0). The inline snapshot projection was not persisting `mt_doc_ordernote` documents.

### Root Cause (Two Issues)

#### Issue #1: Missing `IStartStream` Return Pattern

**What was wrong:**
```csharp
// ❌ INCORRECT - Does NOT persist automatically
public static CreationResponse<Guid> Handle(
    AddOrderNote command,
    HttpContext httpContext,
    IDocumentSession session)
{
    session.Events.StartStream<OrderNote>(noteId, @event);
    return new CreationResponse<Guid>(...);
}
```

**What was needed:**
```csharp
// ✅ CORRECT - Wolverine automatically calls SaveChangesAsync()
public static (CreationResponse<Guid>, IStartStream) Handle(
    AddOrderNote command,
    HttpContext httpContext)
{
    var stream = MartenOps.StartStream<OrderNote>(noteId, @event);
    var response = new CreationResponse<Guid>(...);
    return (response, stream);  // CreationResponse MUST come first
}
```

**Why this matters:**
- Direct `session.Events.StartStream()` usage does NOT enroll in Wolverine's transactional middleware
- Returning `IStartStream` from `MartenOps.StartStream()` signals Wolverine to persist automatically
- Tuple order matters: HTTP response first, side effect (stream creation) second

#### Issue #2: Missing `AutoApplyTransactions` Policy

**What was missing in `Program.cs`:**
```csharp
builder.Host.UseWolverine(opts =>
{
    // ❌ MISSING - Required for Marten integration
    // opts.Policies.AutoApplyTransactions();

    opts.Discovery.IncludeAssembly(...);
});
```

**What was needed:**
```csharp
builder.Host.UseWolverine(opts =>
{
    // ✅ CRITICAL - Auto-apply transactional middleware
    opts.Policies.AutoApplyTransactions();

    opts.Discovery.IncludeAssembly(...);
});
```

**Why this matters:**
- Without `AutoApplyTransactions()`, Wolverine does NOT wrap handlers in transactional middleware
- Marten changes are not automatically committed
- This policy is present in ALL working BCs (Shopping, Orders, Returns, etc.)

### Additional Missing Configuration

**Added `.AddAsyncDaemon(DaemonMode.Solo)` to Marten config** (matching Shopping/Orders pattern):
```csharp
builder.Services.AddMarten(opts => { ... })
    .AddAsyncDaemon(DaemonMode.Solo)  // ✅ Added
    .UseLightweightSessions()
    .IntegrateWithWolverine();
```

**Impact:** While not the root cause, this ensures async projection daemon is configured correctly for future async projections.

### Key Pattern Discovery: Instance Methods Work

**Finding:** ALL working aggregates use **instance** Apply() methods, NOT static methods.

**Examples:**
```csharp
// Cart.cs (Shopping BC) - WORKS
public Cart Apply(ItemAdded @event) => this with { ... };

// Checkout.cs (Orders BC) - WORKS
public Checkout Apply(ShippingAddressProvided @event) => this with { ... };

// Return.cs (Returns BC) - WORKS
public Return Apply(ReturnApproved @event) => this with { ... };
```

**Conclusion:** The skill document `event-sourcing-projections.md` (line 149) states:
> "Apply() methods **MUST be static** for inline projections (thread safety)"

This guidance is **INACCURATE**. Both instance and static Apply() methods work with inline projections. The codebase demonstrates this conclusively across 9 production aggregates.

### Create() Method Patterns

**Both patterns work:**
```csharp
// Pattern A: IEvent<T> wrapper (most common)
public static OrderNote Create(IEvent<OrderNoteAdded> @event) => new(...);

// Pattern B: Raw event type (also valid)
public static Return Create(ReturnRequested @event) => new(...);
```

**Conclusion:** `IEvent<T>` vs raw event type is NOT the issue. Both patterns work correctly.

## 📖 Skill Document Inaccuracies Identified

### 1. event-sourcing-projections.md (Line 149)

**Current guidance:**
> "Apply() methods **MUST be static** for inline projections (thread safety)"

**Reality:** Instance Apply() methods work perfectly. 9 production aggregates prove this.

**Recommended fix:**
```markdown
### Apply() Methods: Instance vs Static

**Current convention:** Instance methods (all production aggregates use this pattern).

Both patterns are valid:

```csharp
// ✅ Instance method (preferred in CritterSupply)
public Cart Apply(ItemAdded @event) =>
    this with { Items = Items.Append(new CartLineItem(...)) };

// ✅ Static method (valid alternative)
public static Cart Apply(Cart current, ItemAdded @event) =>
    current with { Items = current.Items.Append(new CartLineItem(...)) };
```

**Why both work:** Marten's projection system supports both patterns. Instance methods are more concise and idiomatic for immutable records.
```

### 2. marten-event-sourcing.md (Lines 390-412)

**Current guidance:** Says instance methods are "current convention" but lacks emphasis on IStartStream pattern.

**Recommended addition:**
Add explicit section about `IStartStream` return pattern for stream creation handlers:
```markdown
### Creating Event Streams: IStartStream Pattern

**CRITICAL:** When creating new event streams in HTTP endpoints, return `IStartStream` from `MartenOps.StartStream()`:

```csharp
[WolverinePost("/api/carts")]
public static (CreationResponse<Guid>, IStartStream) Handle(InitializeCart command)
{
    var cartId = Guid.CreateVersion7();
    var @event = new CartInitialized(...);

    // CRITICAL: Use MartenOps.StartStream() and return IStartStream
    var stream = MartenOps.StartStream<Cart>(cartId, @event);
    var response = new CreationResponse<Guid>(...);

    return (response, stream);  // CreationResponse MUST come first
}
```

**Anti-pattern (does NOT persist):**
```csharp
// ❌ DOES NOT WORK - Wolverine won't persist automatically
public static CreationResponse<Guid> Handle(
    InitializeCart command,
    IDocumentSession session)
{
    session.Events.StartStream<Cart>(cartId, @event);
    return new CreationResponse<Guid>(...);
}
```
```

## 💡 Lessons Learned

### For AI Agents

1. **Code is source of truth** - When documentation conflicts with working code, trust the code
2. **Survey ALL examples** - Don't rely on single examples; examine all instances of a pattern
3. **Return patterns matter** - `IStartStream` return type is NOT optional for stream creation
4. **Wolverine policies are critical** - `AutoApplyTransactions()` is required, not suggested
5. **Test failures have multiple causes** - Projection issue vs authorization issue vs test infrastructure issue

### For Skill Documents

1. **Static vs instance Apply() methods** - Current guidance is inaccurate; both work
2. **IStartStream pattern** - Needs prominent documentation with anti-pattern warnings
3. **Wolverine transactional policies** - Needs explicit "REQUIRED" emphasis, not "recommended"
4. **Examples should match reality** - Update skill files to reflect actual codebase patterns

### For CritterSupply Patterns

1. **HTTP endpoint stream creation** - Always use `(CreationResponse, IStartStream)` tuple return
2. **Program.cs baseline** - All BCs should have `AutoApplyTransactions()`, `AddAsyncDaemon()`, and `IntegrateWithWolverine()`
3. **Test fixtures** - Authentication handlers need stable identities across requests in same test

## Key Decisions

### 1. Renamed Real-Time Events to Avoid Naming Conflicts
**Problem:** `OrderPlaced` conflicted with `Messages.Contracts.Orders.OrderPlaced`

**Solution:** Renamed to domain-specific names:
- `ActiveOrderIncremented` (instead of OrderPlaced)
- `ActiveOrderDecremented` (instead of OrderFulfilled)
- `PendingReturnIncremented` (instead of ReturnRequested)

**Rationale:** Clearer intent, avoids namespace pollution, makes hub messages distinct from integration messages.

### 2. Stub Endpoints for Fast Iteration
**Decision:** Return hardcoded data from HTTP endpoints (no Marten projections yet)

**Rationale:**
- Enables immediate testing of WASM → API → WASM round trip
- Decouples frontend work from projection implementation
- Allows parallel work on projections in Phase 3

**Next Step:** Replace stubs with real Marten projections in Session 8+

### 3. Updated LiveMetricUpdated Schema
**Problem:** Existing `LiveMetricUpdated` had `OrderCount`, `Revenue`, `PaymentFailureRate` — didn't match Dashboard KPI cards

**Solution:** Changed to:
- `ActiveOrders`, `PendingReturns`, `LowStockAlerts`, `TodaysRevenue`

**Impact:** Had to update `OrderPlacedHandler` to return stub values for PendingReturns and LowStockAlerts

## Challenges & Lessons

### Challenge 1: MudBlazor Type Parameters
**Issue:** MudChip requires T="string" in MudBlazor v9+

**Resolution:** Added T="string" to all MudChip components

**Lesson Learned:** Always build after adding UI components (don't batch too many changes)

### Challenge 2: Naming Conflicts with Integration Messages
**Issue:** `OrderPlaced` existed as both:
- Integration message: `Messages.Contracts.Orders.OrderPlaced`
- Real-time event: `Backoffice.RealTime.OrderPlaced`

**Resolution:** Renamed real-time events to intent-based names (ActiveOrderIncremented, etc.)

**Lesson Learned:** Use domain-specific names for hub events to avoid namespace collisions

### Challenge 3: Updating Existing Handlers
**Issue:** Changing `LiveMetricUpdated` schema broke `OrderPlacedHandler`

**Resolution:** Updated handler to use new parameter names + stub values

**Lesson Learned:** Search for usages before changing message contracts

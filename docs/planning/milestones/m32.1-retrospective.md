## Key Technical Wins

### W1: Tiered Timeout Strategy for WASM E2E Tests

**Challenge:** Blazor WASM cold start (5-30s) + MudBlazor initialization + JWT auth + SignalR connection compound to create 30-40s total delay. Fixed 30s timeout too aggressive for initial load but wasteful for fast operations.

**Solution:** Tiered timeout strategy optimized for actual operation timing:
- Initial page load: 15s + explicit MudBlazor provider check
- Authenticated navigation: 15s (auth state propagation needs time)
- Element visibility: 10s (standard Playwright wait)
- SignalR connection: 15s (JWT auth must complete first)
- State checks: 5s (DOM state polling)

**Why It Works:**
- Explicit MudBlazor hydration check (`.mud-dialog-provider` CSS class) ensures components ready
- Increased dashboard navigation timeout accounts for auth state propagation (1-3s delay)
- SignalR timeout accounts for JWT dependency + retry attempts

**Pattern:**
```csharp
// Step 1: Wait for network idle
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);

// Step 2: Wait for MudBlazor initialization
await _page.WaitForSelectorAsync(".mud-dialog-provider", new() { Timeout = 15_000 });

// Step 3: Wait for specific element
await EmailInput.WaitForAsync(new() { Timeout = 15_000 });
```

**Reusability:** Pattern documented in Session 16 retrospective. Will be added to `docs/skills/e2e-playwright-testing.md` for future WASM E2E tests.

---

### W2: Explicit Hydration Detection Prevents Flaky Tests

**Challenge:** `NetworkIdle` doesn't guarantee MudBlazor components are ready. Tests were clicking elements before MudBlazor event handlers attached.

**Solution:** Check for framework-specific markers (MudBlazor provider classes, SignalR connection state, auth state) in addition to `NetworkIdle`.

**Why It Works:**
- MudBlazor v9+ creates `.mud-dialog-provider` and `.mud-snackbar-provider` containers during initialization
- If provider exists, MudBlazor is ready (providers are last step of initialization)
- SignalR connection indicator (`realtime-connected`) proves hub connection established
- URL contains `/dashboard` proves auth state propagated and navigation completed

**Impact:** Eliminates race conditions where tests interact with UI before WASM/MudBlazor/JWT/SignalR fully initialized.

---

### W3: Test-ID Naming Conventions Prevent Refactoring Breakage

**Challenge:** Session 14 found 17 test-id mismatches between Page Object Models and Razor components. Inconsistent naming (`kpi-active-orders` vs `kpi-total-orders`, `customer-search-btn` vs `nav-customer-service`) caused test failures even though functionality worked.

**Solution:** Document test-id conventions early with clear patterns, examples, and anti-patterns.

**Convention:**
- **Semantic names:** Describe what element represents, not how it looks
- **Kebab-case:** Aligns with HTML attribute conventions
- **Hierarchical structure:** Parent-child relationships (KPI card contains `kpi-value`)
- **Stable identifiers:** Don't change when UI styling or ordering changes

**Why It Works:**
- Page Object Models define contract (expected test-ids)
- Razor components fulfill contract (implement those test-ids)
- Conventions prevent mismatches during implementation
- Refactoring UI styling/layout doesn't break tests

**Reusability:** Conventions documented in Session 15 retrospective. Will be added to `docs/skills/e2e-playwright-testing.md`.

### W5: In-Memory JWT Storage Pattern Scales to Third BFF

**Challenge:** Vendor Portal (M22) established in-memory JWT storage pattern (not localStorage — XSS risk). Backoffice is third BFF using this pattern. Pattern must scale without modification.

**Solution:** Vendor Portal pattern copied verbatim:

- `BackofficeAuthStateProvider.cs` (custom `AuthenticationStateProvider`)

**Why It Works:**

## Critical Lessons Learned

### L1: WASM E2E Tests Need Tiered Timeouts

**Discovery:** Blazor WASM cold start (5-30s) + MudBlazor initialization (2-5s) + JWT auth (1-3s) + SignalR connection (1-5s) compound to create 30-40s total delay.

**Impact:** Fixed 30s timeout was too aggressive for initial load but wasteful for fast operations. Tests timed out on first page load.

**Solution:** Use tiered timeout strategy optimized for actual operation timing (15s initial, 15s navigation, 10s elements, 5s state checks).

**Pattern:**
```csharp
// Initial page load (15s + MudBlazor check)
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);
await _page.WaitForSelectorAsync(".mud-dialog-provider", new() { Timeout = 15_000 });

// Authenticated navigation (15s for auth propagation)
await _page.WaitForURLAsync(url => url.Contains("/dashboard"), new() { Timeout = 15_000 });

// Element visibility (10s standard)
await TotalOrdersCard.WaitForAsync(new() { Timeout = 10_000 });

// State checks (5s DOM polling)
await RealtimeIndicator.WaitForAsync(new() { Timeout = 5_000 });
```

**Recommendation:** Document in `docs/skills/e2e-playwright-testing.md` Section: "WASM Timeout Tuning".

### L3: Page Object Models Should Be Written Before Components

**Discovery:** Session 14 found 17 test-id mismatches because Dashboard.razor was written first, then DashboardPage.cs expected different test-ids.

**Impact:** Tests failed not because functionality broke, but because test-ids didn't match POM locators.

**Better Approach:**
1. Write Gherkin `.feature` file (user stories)
2. Write Page Object Model with expected test-ids (defines contract)
3. Write Razor component implementing those test-ids (fulfills contract)
4. Write step definitions using Page Object Model

**Why:** POM defines the contract. Components fulfill the contract. If component is written first, POM must adapt to component's arbitrary test-ids.

**Recommendation:** Document in `docs/skills/e2e-playwright-testing.md` Section: "Page Object Model Best Practices".

---

### L4: Auth State Propagation Needs Extra Time in WASM

**Discovery:** After successful login API call, Blazor WASM takes 1-3s to:
1. Update `BackofficeAuthState`
2. Fire `OnChange` event
3. Call `NotifyAuthenticationStateChanged()`
4. Re-render protected components
5. Execute `NavigationManager.NavigateTo("/dashboard")`

**Impact:** 10s timeout for dashboard navigation was insufficient. 15s is more reliable.

**Pattern:**
```csharp
await LoginButton.ClickAsync();

// Wait for auth state propagation + navigation (15s instead of 10s)
await _page.WaitForURLAsync(url => url.Contains("/dashboard"), new() { Timeout = 15_000 });
```

**Why:** Auth state propagation is asynchronous. Protected routes check auth state before allowing navigation. If auth state not updated yet, navigation blocks until state updates.

**Recommendation:** All WASM E2E tests with authentication should use 15s timeout for post-login navigation.

---

### L5: SignalR Connection Depends on JWT Auth Completion

**Discovery:** SignalR hub uses `AccessTokenProvider` delegate to get JWT from `BackofficeAuthState`. If auth state isn't ready, connection fails silently and retries.

**Impact:** SignalR connection timeout must be >= auth state propagation timeout. 10s was insufficient because auth state took 1-3s to propagate, leaving only 7-9s for SignalR connection + retries.

**Solution:** Increase SignalR connection timeout to 15s (same as auth propagation timeout).

**Pattern:**
```csharp
// After login, wait for auth state propagation
await _page.WaitForURLAsync(url => url.Contains("/dashboard"), new() { Timeout = 15_000 });

// Then wait for SignalR connection (15s to allow for retry attempts)
await RealtimeIndicator.WaitForAsync(new() { Timeout = 15_000 });
```

**Why:** SignalR client retries connection on failure. If JWT isn't ready on first attempt, retry succeeds after auth state propagates. 15s timeout allows for 2-3 retry attempts.

**Recommendation:** All WASM E2E tests with SignalR should use 15s timeout for connection establishment.

### R5: Consider Playwright's `page.waitForLoadState('domcontentloaded')`

**Why:** Current pattern uses `NetworkIdle` but doesn't wait for `DOMContentLoaded`. This may cause timing issues.

**Pattern:**
```csharp
// Current
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);

// Enhanced
await _page.WaitForLoadStateAsync(LoadState.DOMContentLoaded);
await _page.WaitForLoadStateAsync(LoadState.NetworkIdle);
```

**Rationale:** `DOMContentLoaded` fires when HTML is parsed but before all resources load. `NetworkIdle` fires when network activity stops. Waiting for both ensures DOM is ready AND resources loaded.

**Impact:** May reduce flaky tests caused by clicking elements before DOM is ready.

**Estimated Effort:** 5-10 minutes per Page Object Model (trivial change)

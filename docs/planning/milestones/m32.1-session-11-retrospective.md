### 4. Critical Bug Fix: FindWasmRoot() Path Issue ✅

**Root Cause Identified:**
`FindWasmRoot()` was returning `src/Backoffice/Backoffice.Web/wwwroot` which only contained:
- `index.html`
- `appsettings.json`

**Missing:** The `_framework/` directory with all compiled Blazor WASM files (`.dll`, `.wasm`, `.dat`, `blazor.webassembly.js`)

**Why This Caused All Failures:**
1. Browser loads `index.html` successfully (200 OK)
2. `index.html` references `_framework/blazor.webassembly.js` → **404 NOT FOUND**
3. Blazor runtime never initializes
4. `appsettings.json` never requested (Blazor bootstrap failed)
5. Login form never renders (WASM not hydrated)
6. Tests timeout waiting for elements that never appear

**Fix Applied:**
```csharp
// OLD: Checked src/wwwroot first (wrong!)
var candidate = Path.Combine(current, "src", "Backoffice", "Backoffice.Web", "wwwroot");
if (Directory.Exists(candidate))
    return candidate; // Returns immediately, misses bin output

// NEW: Check bin output first, verify _framework exists
var binWwwroot = Path.Combine(current, "src", "Backoffice", "Backoffice.Web", "bin", "Debug", "net10.0", "wwwroot");
if (Directory.Exists(binWwwroot) && Directory.Exists(Path.Combine(binWwwroot, "_framework")))
    return binWwwroot; // ✅ Correct location with compiled files
```

**Verification:**
```bash
$ ls src/Backoffice/Backoffice.Web/bin/Debug/net10.0/wwwroot/
_framework/  (36,864 bytes - has all WASM files)

$ ls src/Backoffice/Backoffice.Web/wwwroot/
index.html  appsettings.json  (only 2 files)
```

### 5. Directory Traversal Permission Fix ✅

**Secondary Bug:** Initial fix used `GetDirectories(current, "Backoffice.Web", SearchOption.AllDirectories)` which triggered:
```
System.UnauthorizedAccessException: Access to the path '/home/packer' is denied.
```

**Solution:** Walk up directory tree explicitly to find repo root, then construct path directly:
```csharp
// Walk up to find repo root (has src/ directory)
while (current != null)
{
    var srcDir = Path.Combine(current, "src");
    if (Directory.Exists(srcDir))
    {
        // Construct explicit path (no recursive traversal)
        var binWwwroot = Path.Combine(current, "src", "Backoffice", "Backoffice.Web",
            "bin", "Debug", "net10.0", "wwwroot");
        // ...
    }
    current = Directory.GetParent(current)?.FullName;
}
```

**Result:** No more UnauthorizedAccessException, clean path resolution

## 🎓 Key Lessons Learned

### L1: Blazor WASM Requires Build Output, Not Source Files

**Lesson:** Blazor WASM `wwwroot` in source (`src/*/wwwroot`) contains only static assets. The compiled `_framework/` directory with `.dll`, `.wasm`, and `blazor.webassembly.js` files is generated during build and placed in `bin/Debug/net10.0/wwwroot`.

**Why This Matters:**
- E2E tests must serve from build output, not source
- This is different from Blazor Server (which compiles on server)
- Missing `_framework/` causes silent bootstrap failures

**Pattern for Future:**
```csharp
// ALWAYS verify _framework exists when locating WASM wwwroot
if (Directory.Exists(wwwroot) && Directory.Exists(Path.Combine(wwwroot, "_framework")))
    return wwwroot; // ✅ Has compiled files
```

### L2: Playwright Tracing is Essential for E2E Debugging

**Lesson:** Without tracing, E2E test failures are opaque black boxes. Browser console errors alone don't show screenshots, DOM state, or network traffic.

**Why This Matters:**
- Timeout errors provide zero diagnostic information
- Trace files show exact DOM state when test failed
- Screenshots reveal visual issues (loading spinners stuck, blank pages)
- Network tab shows 404s, CORS errors, failed API calls with URLs

**Pattern for All E2E Tests:**
```csharp
// BeforeScenario
await context.Tracing.StartAsync(new() { Screenshots = true, Snapshots = true, Sources = true });

// AfterScenario (on failure)
if (scenarioContext.ScenarioExecutionStatus == ScenarioExecutionStatus.TestError)
{
    await context.Tracing.StopAsync(new() { Path = $"traces/{scenario}.zip" });
    Console.WriteLine($"View with: pwsh bin/Debug/net10.0/playwright.ps1 show-trace traces/{scenario}.zip");
}
```

### L3: SearchOption.AllDirectories is Dangerous in CI

**Lesson:** `Directory.GetDirectories(path, pattern, SearchOption.AllDirectories)` traverses the ENTIRE filesystem tree, triggering permission errors on protected directories like `/home/packer`, `/root`, `/sys`.

**Why This Matters:**
- CI environments have restricted filesystem access
- Recursive traversal is slow and unnecessary
- Explicit path construction is faster and safer

**Pattern for Finding Files:**
```csharp
// BAD: Recursive traversal (slow, permission errors)
var files = Directory.GetDirectories(root, "MyDir", SearchOption.AllDirectories);

// GOOD: Walk up parent directories explicitly
var current = AppContext.BaseDirectory;
while (current != null)
{
    var candidate = Path.Combine(current, "expected", "path");
    if (Directory.Exists(candidate))
        return candidate;
    current = Directory.GetParent(current)?.FullName;
}
```

### L4: Browser Console Logging Reveals JavaScript Errors

**Lesson:** Blazor WASM failures often manifest as JavaScript errors in browser console, not HTTP errors. Console logging is critical for diagnosing WASM hydration failures.

**Why This Matters:**
- 404s on `blazor.webassembly.js` show in console, not network logs
- JavaScript exceptions during WASM initialization are silent without console logging
- CORS errors show in console with detailed origin mismatch info

**Pattern:**
```csharp
page.Console += (_, msg) =>
{
    var level = msg.Type switch
    {
        "error" => "ERROR",
        "warning" => "WARN",
        _ => "INFO"
    };
    Console.WriteLine($"[Browser Console {level}] {msg.Text}");
};
```

### L5: Diagnostic Logging Before Anything Else

**Lesson:** When debugging E2E tests, add diagnostic logging FIRST before attempting fixes. You can't fix what you can't see.

**Why This Matters:**
- Session 10: No logging → 32 timeouts, zero diagnostic info
- Session 11: Added logging → immediately identified wwwroot path issue
- Logging shows which code paths are executing and which are never reached

**Pattern:**
```csharp
// Log inputs at boundaries
Console.WriteLine($"✅ [ComponentName] Operation started: {input}");

// Log successful outcomes
Console.WriteLine($"✅ [ComponentName] Operation succeeded: {result}");

// Log failures with context
Console.WriteLine($"❌ [ComponentName] Operation failed: {error}. Input was: {input}");
```

### 2. Consistent Helper Function Pattern

**Implementation:**
```csharp
private static string FormatTimeSince(DateTimeOffset timestamp)
{
    var elapsed = DateTimeOffset.UtcNow - timestamp;

    if (elapsed.TotalSeconds < 60)
        return "just now";
    if (elapsed.TotalMinutes < 60)
        return $"{(int)elapsed.TotalMinutes} minute{((int)elapsed.TotalMinutes == 1 ? "" : "s")} ago";
    if (elapsed.TotalHours < 24)
        return $"{(int)elapsed.TotalHours} hour{((int)elapsed.TotalHours == 1 ? "" : "s")} ago";

    return timestamp.ToLocalTime().ToString("yyyy-MM-dd HH:mm");
}
```

**Why it worked:**
- Applied to both Dashboard.razor and Alerts.razor for consistent UX
- Simple, pure function with no dependencies
- Meets P1-2 acceptance criteria ("communicate 'updating' vs 'stale' clearly")

**Pattern to reuse:** For cross-page UI utilities in Blazor WASM, duplicate simple helper functions rather than creating shared utility classes (avoids premature abstraction).

## What Could Be Improved

### 1. Backend DTO Design Inconsistency

**Issue:** Backend DTOs sometimes include timestamps (`QueriedAt`) and sometimes don't.

**Example:**
- `AlertFeedResponse` — **had to add** `QueriedAt` in Session 3
- `DashboardMetrics` — **already had** `GeneratedAt` from Session 1

**Root cause:** No established pattern for "when to include backend query timestamps in DTOs."

**Proposed pattern:**
```markdown
## DTO Timestamp Convention

All backend query DTOs SHOULD include a timestamp field (e.g., `QueriedAt`, `GeneratedAt`) to support:
1. Frontend "last refreshed" indicators (P1-2 acceptance criteria)
2. Cache staleness detection
3. Debug/troubleshooting (when was this data generated?)

**Exceptions:**
- Write-only commands (POST/PUT/DELETE responses with no read model)
- Streaming/event-based responses (timestamp per event, not per response envelope)
```

**Skill file candidate:** Consider adding to `docs/skills/blazor-wasm-jwt.md` or new file `docs/skills/dto-design-patterns.md`.

### 2. Helper Function Duplication

**Issue:** `FormatTimeSince()` helper duplicated in Dashboard.razor and Alerts.razor.

**Why we did it:** Following CritterSupply pattern of avoiding premature abstraction.

**When to refactor:** After 3rd usage (CustomerSearch page adds timestamp display) → extract to shared `TimeFormatting.cs` utility class.

**Future consideration:** If M32.3 adds more timestamp-heavy pages, create `src/Backoffice/Backoffice.Web/Helpers/TimeFormatting.cs`.

## Learnings for Future Sessions

### 1. Terminology Audits Are Fast

**Observation:** P2-9 terminology review took < 5 minutes by reading three page files.

**Lesson:** Don't defer terminology audits — they're quick wins and prevent inconsistency debt.

### 2. Empty State UX Is Hard to Retrofit

**Observation:** Alerts and CustomerSearch already had excellent empty states because they were considered during initial implementation.

**Lesson:** Always implement empty state UX *during* feature development, not as a "P2 hardening" task. Retrofitting requires more context switching.

### 3. Backend-First Pattern for Data Freshness

**Observation:** Backend `QueriedAt`/`GeneratedAt` timestamps are better than frontend `DateTimeOffset.UtcNow` for cache control.

**Lesson:** When adding "last refreshed" indicators, prefer backend timestamps (enables server-side cache headers, CDN integration, multi-client sync).
